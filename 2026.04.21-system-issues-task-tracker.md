## Task 01: Inability to Edit Member Details in Group View (Role Validation Error)

- Status: Closed

### Description
When a VSLA Facilitator opens My Groups and edits an existing group member, updates fail due to a validation error requiring the role field.

### Steps to Reproduce
1. Log in as a VSLA Facilitator.
2. Navigate to VSLA Facilitator > My Groups.
3. Open any group.
4. Attempt to edit an existing member record.

### Observed Behaviour
A validation error appears: "The role field is required," and the member update is rejected.

### Expected Behaviour
Users should be able to update member details without role-related validation failures unless role is intentionally mandatory and correctly presented in the form.

### Developer Remarks
- Root cause confirmed: mobile `PUT` helper sent `multipart/form-data` (`FormData`) for `PUT` requests. For this stack, payload could arrive empty on server for `PUT`, causing role endpoint validation to fail with "The role field is required."
- Endpoint path verified:
	- Role update: `PUT /api/agent-vsla/members/{id}/role` requires `role`.
	- Member edit: `PUT /api/agent-vsla/members/{id}` does not require `role`.
- Fix implemented (mobile): changed `http_put` to send JSON map payload (not `FormData`) and stopped forced overwrite of caller `user_id`.
- Live verification (real endpoint):
	- `multipart PUT` to role endpoint returned: `The role field is required.`
	- `JSON PUT` to same endpoint succeeded: `Role updated successfully`.
- Dummy validation rerun (2026-04-21) passed end-to-end:
	- Baseline role/phone captured.
	- Negative cases failed as expected.
	- Positive JSON cases succeeded.
	- Final state reverted and verified to original values.
- Files updated:
	- `/Users/mac/Desktop/github/fao-ffs-mis-mobo/lib/utils/Utils.dart`
